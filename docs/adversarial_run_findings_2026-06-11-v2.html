5. Real findings, ranked

High Audit log is not capturing security events

Four scenarios produced zero new rows in the audit-log DDB table within the 5-second post-trigger window:

  • 6 failed Cognito sign-in attempts (rate-limit event)
  • Cross-persona request against a CISO-only route
  • Forged-claim token request
  • Legitimate CISO action approve

Why this matters for the demo AuditLogs.jsx is a CISO/GRC-visible page and is part of the demo's governance narrative. With nothing being written, the page shows only the rows scripts/seed_mock_data.py seeded at deploy time. A customer walkthrough hitting "show me the audit log of what just happened" sees nothing.

Fix direction Add a boto3.client("dynamodb").put_item in api_handler.py at three points: any 4xx from _caller_user_id resolution, any /actions/{id}/<transition> handler entry, and a Cognito EventBridge subscription on the signIn failure event.

High Cognito brute-force throttle did not engage

The harness fired 10 rapid InitiateAuth calls with a wrong password against a non-existent username. Cognito should respond with LimitExceededException after roughly 5 attempts. All 10 returned NotAuthorizedException. No throttle observed.

Operational note Cognito's default policy does throttle. The fact that this account does not points at an explicit configuration. Worth checking AccountTakeoverRiskConfiguration on the user pool.

Medium Workflow bypass: state-machine transitions out of terminal states

Two of the four action-state-machine probes failed:

  • logic.workflow.reject-after-execute: after an action was executed, the harness was able to reject it. Returned 200.
  • logic.workflow.escalate-from-terminal: after an action was rejected, the harness was able to escalate it. Returned 200.

Why this matters for the demo The action lifecycle in ActionCenter.jsx assumes terminal states are terminal. If a customer walkthrough demonstrates approve-then-execute-then-celebrate and a curious viewer hits reject, the demo's claim of an immutable audit trail breaks.

Fix direction In _handle_action_transition: add a precondition check that the action's current state field permits the requested transition. Reject otherwise with 409.

Medium Race condition on concurrent approve

Three concurrent POST /actions/{id}/approve on the same action all returned 200. The handler does not use a conditional write.

Fix direction Add ConditionExpression="state = :expected" on the approve write in _handle_action_transition. First concurrent attempt satisfies the condition; the rest get ConditionalCheckFailedException and map to 409.

Medium Excessive data exposure on GET /actions

All 4 personas hitting GET /actions get back JSON that contains a field flagged by the field-exposure walker. Same for GET /token-usage as CISO. The walker matches on internal IDs (_internal, internal_id) or cross-user emails.

Fix direction In the response shaping inside _handle_get_actions and _handle_get_token_usage, project only the fields the SPA actually consumes. The full DDB row should not be returned. Add a shared _strip_internal helper.

Medium Missing or weak operational guardrails: TLS, headers, rate limit

Same theme as yesterday; the count is down (32 fails became 22), but the items are still:

  • TLS 1.0 and 1.1 still accepted by CloudFront (2 probes).
  • CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy missing on 4 routes each.
  • HSTS missing on 3 hosts probed.
  • 4 GET routes (/findings, /conversations, /dashboard, /agent-status) accept 20 RPS for 5 s with no 429.

Fix direction One CloudFront Response Headers Policy attached to the distribution + minimum-TLS to TLSv1.2_2021 closes most of these. One WAF rate-based rule at 200 req/min per IP closes the rate-limit findings. Both console-only.

Medium Cross-pool JWT still accepted

The fault layer's cross-pool JWT probe still passes the gate at GET /token-usage. Today the test reports it as fault.error-prop.cross-pool-jwt.cloudwatch-logged failing LOW (CloudWatch wasn't logging it either), which surfaces the same underlying issue: _caller_claims does not validate the aud claim.

Fix direction In _caller_claims: assert claims.get("aud") == COGNITO_CLIENT_ID.
