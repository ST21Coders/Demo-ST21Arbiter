## 1. Architecture overview

We introduce a new top-level directory `tests-adversarial/` that owns the full harness end to end. It is separate from `tests-e2e/` because the existing Playwright suite is wired for local mock mode against `http://localhost:5173` (see `tests-e2e/playwright.config.ts` lines 3-4 and the `webServer` block lines 29-39), and the spec mandates testing the deployed dev CloudFront. Mixing the two would either dilute the existing mock-mode coverage or force `tests-e2e/` to grow live-mode branches that are not in its remit. Keeping them apart also lets the harness own its own dependency closure (Playwright config, hypothesis, garak/PyRIT) without polluting the SPA test layer.

The harness is organized as four independent layers under `tests-adversarial/`: `e2e/` (Playwright JS, matching the JS-only project rule in CLAUDE.md), `fuzz/` (pytest + hypothesis, matching the existing `tests/security/` pattern), `auth/` (pytest, sharing the JWT-forging helpers already in `tests/conftest.py::make_jwt`), and `llm/` (pytest, with hand-rolled probe orchestration that imports garak prompt templates as data only — not as a runner — to keep the dependency footprint small). Each layer is runnable standalone via `npm run test:<layer>`, and a single Python orchestrator (`scripts/run_all.py`) drives all four in sequence for `npm run test:all`. Python wins over Node for the orchestrator because three of four layers are Python and the orchestrator must read pytest JSON output, tally Bedrock cost from a Python pricing source, and write a unified report — all of which is cleaner in Python than shelling out from Node.

The reporting subsystem is a Python module (`src/reporting/`) that consumes per-layer JSON outputs and emits `report.html`, `report.json`, and `summary.md` into `test-reports/<UTC-ISO-timestamp>/`. HTML is hand-rolled with a single Jinja2 template (already a transitive boto3 dep, no new install) so the report is self-contained and forwardable without external assets. The coverage matrix is built from a hand-curated `coverage/manifest.json` rather than auto-extracted from source — the spec emphasizes "auditable" completeness, and a manifest is far more reviewable than a parser's output. A drift-check script (`scripts/check_manifest_drift.py`) reconciles the manifest against `ui/src/pages/`, `api_handler.py`, and `agents/*/agent.py` at every run start and fails fast if any surface is missing. This gives us auto-detection of drift with manual control over the matrix.

```
+-------------------------------------------------------------+
|                  scripts/run_all.py (Python)                |
|   pre-flight  ->  e2e  ->  fuzz  ->  auth  ->  llm  ->  report
+----+--------------+--------+--------+--------+--------+-----+
     |              |        |        |        |        |
     v              v        v        v        v        v
  pricing.py     PW(JS)  pytest   pytest   pytest   renderer.py
   tracker.py    json    json     json     json     -> html/md/json
   manifest      out     out      out      out
                  \______________ test-reports/<ts>/ ______________/
```

Deviations from the prompt sketch:

- `pyproject.toml` over `requirements.txt`. The harness has multiple entry points (`run_all.py`, `promote_baseline.py`, `check_manifest_drift.py`) and dev-vs-test deps split cleanly via `[project.optional-dependencies]`. The existing project root has no `pyproject.toml` so we are not displacing one.
- `src/reporting/templates/*.j2` (Jinja2 files) added because hand-rolling HTML in Python strings is unreadable for a 60-cell matrix.
- Per-layer `corpus/` folders sit inside each layer rather than under a top-level `corpus/` so each layer is self-contained when run standalone.

## 3. Key technical decisions

Each non-obvious choice with the rejected alternative.

**Why `tests-adversarial/` and not `tests-e2e/` extension.** `tests-e2e/` is mock-mode-default and aimed at local-loop dev signal (`webServer` autostarts Vite, fixture injects a CISO-only fake token). Bolting a deployed-env layer onto it would either flip the default behavior (and break the existing dev signal) or grow a parallel branch that is harder to reason about. Separate directory, separate `package.json`, no shared `playwright.config`.

**Why a Python orchestrator instead of Node.** Three of four layers are Python (fuzz, auth, llm), the pricing source of truth is Python (`agents/_shared/token_usage.py`), and the orchestrator needs to read pytest JSON, decimal-accurate cost math, and a Jinja2 template. Doing this in Node means shelling out, JSON re-encoding, and a second pricing copy. Python is one fewer language boundary. The Playwright layer still uses Node — the orchestrator calls `npm test` and reads the JSON reporter output.

**How Playwright runs against the deployed Cognito-gated app.** Programmatic login via `boto3 cognito-idp.InitiateAuth` with `USER_PASSWORD_AUTH` for each of the 4 demo users (using `DEMO_PASSWORD` from the local shell). The returned `IdToken` is injected into Playwright's `storageState` via `page.addInitScript` writing to `sessionStorage` under key `arbiter.tokens` (matching the SPA's expected shape — see `tests-e2e/fixtures.ts:35-40`). This bypasses the Cognito Hosted UI for the bulk of E2E coverage. A single dedicated test (`cognito-hosted-ui.spec.js`) does exercise the real Hosted UI to cover AC18. Each persona's `storageState` is cached to a tmp file under `e2e/storage-states/` (gitignored) for the duration of the run, then deleted in global teardown.

**Why programmatic login dodges the React StrictMode double-fire risk.** The `handleCallback` in-flight guard (CLAUDE.local.md, `ui/src/hooks/useAuth.js`) is the canonical pattern for the OAuth callback path. By skipping the Hosted UI callback for non-auth tests, we never trigger that path twice in the same render and the StrictMode risk is moot. The single Hosted UI test signs in once per run, watches for the callback URL transition once, and asserts the post-callback state — same pattern the SPA uses in real use.

**How fuzz layer avoids destructive writes to deployed dev DDB.** Two layers of guardrails: (1) every fuzz request body is rate-limited to one per route-shape per second to dodge accidental burst writes, and (2) any test that creates a row (e.g. `/actions` POST mass-assignment) records the created id and deletes it in the test's teardown via the existing `DELETE /conversations/{id}` or by direct DDB delete using the test identity's permissions. Body size is capped at 1MB (the curated "oversized" payload) — we do not generate hypothesis examples larger than that. The chat-with-no-signature path is documented-unsafe per AC11 and we record its existence but never assert remediation.

**How the coverage matrix gets its inventory.** Hand-curated `coverage/manifest.json`. Three top-level keys: `pages`, `routes`, `tools`, each an array of objects with stable ids. Drift detector (`scripts/check_manifest_drift.py`) runs at the start of every `run_all.py`. It:
- Globs `ui/src/pages/*.jsx` and asserts every file is referenced by exactly one `pages[].file`.
- Greps `Infra/functions/api_handler/api_handler.py` for `if path == "/..."` blocks between lines 133 and 216 and asserts every literal route is in `routes[].path`.
- Greps `agents/*/agent.py` for `@tool` decorators and asserts every decorated function name is in `tools[].name`.
On drift, the run exits non-zero with `MANIFEST_DRIFT: pages added/removed: [...]`. Editing the manifest is a deliberate act, and a CODEOWNERS-style review point.

**How the LLM probe cap (30 per run) is enforced.** A counter in `cost/tracker.py::increment_probe()` that all LLM tests must call via a pytest fixture (`llm_probe_budget`). Fixture raises `pytest.skip("probe budget exhausted")` once the counter hits 30. CLI override is `--llm-probes <n>` on `run_all.py`, propagated to the LLM pytest via `--llm-probes` (custom flag defined in `llm/conftest.py`).

**Where transcripts and screenshots go.** Each layer writes to its own subdirectory in `test-reports/<timestamp>/<layer>/`. Playwright `screenshot: 'only-on-failure'` (already the pattern in `tests-e2e/playwright.config.ts:21`) lands under `e2e/test-results/`. Fuzz writes `transcripts/<route>.jsonl` (one JSONL row per request+response pair). Auth writes `probes.jsonl` (one row per probe). LLM writes `transcripts/<probe-id>.jsonl` (prompt + response). The reporter writes relative-path links into `report.html` and `summary.md` so the report directory is self-contained when forwarded.

**AC11 `auth.chat.no-signature` is documented-unsafe, not a fix-me.** The test records the platform's behavior (currently: signature ignored, 200 returned). The pass criterion is `expected: "documented-unsafe"` matched against the actual outcome. If the platform tightens and starts rejecting unsigned tokens, the test flips status to `regression` with severity:medium, because legitimate callers will also break. Implementer must understand this is a regression detector, not a "fix me." This is encoded in the test's expected_outcome field, not in code comments.

**AC23 pricing drift mechanism.** `src/cost/pricing.py::load_pricing()` does two reads: (1) imports `agents._shared.token_usage.MODEL_PRICING` directly (the harness's pyproject adds `agents/` to PYTHONPATH at install time, no copy), and (2) parses `ui/src/mockData.js::MODEL_PRICING` with a small regex extractor that returns `{model_id: {input, output}}`. If the two dicts disagree on any model's input or output rate, `PricingDriftError` is raised before any network call. This is the AC23 mechanism. Both files remain off-limits per the project rules — the harness reads, never writes.

**Why we keep `MODEL_PRICING` off-limits even though we read two copies.** CLAUDE.md flags this constant as duplicated-by-design. Our harness does not "fix" the duplication; it consumes both and catches drift. Updating prices remains an explicit two-file commit done by whoever adds a model, not by us.

## 6. Risks

1. **Polluting demo DDB tables.** Largest blast-radius item. Mitigation: every test-created session tags its prompt with `[harness]` so demo users can identify them. Tests that POST `/actions` delete the row in teardown. Auth layer's forged-groups tests use read-only endpoints (`GET /token-usage`) where possible.
2. **Master orchestrator silently flipped to Claude Sonnet 4.6.** Per CLAUDE.local.md the live master uses Claude Sonnet 4.6 (10-30x cost over Nova 2 Lite). Pre-flight reads the deployed master's `MODEL_ID` via a "what model are you?" probe (best-effort, single turn ~$0.005) and re-validates the estimate against the live model. If live model is Claude, the LLM layer's 30-probe cap stays but the cost estimate jumps roughly $0.15 -> $4.50, so a $1 cap forces the layer to skip with a `severity:info` notice. The implementer must not "fix" this by raising the cap — the cap is the contract.
3. **`lookup_url_category` triggers a live ZIA API.** Limit to one invocation per run (the Zscaler tool probe in `llm/corpus/probes.yaml`). The URL probed is fixed and logged.
4. **Bedrock concurrency throttle hits mid-run.** LLM probes are sent serially, not in parallel (`pytest -p no:xdist` for the llm dir). Adds wall-clock but stays under the 10-minute budget per AC24.
5. **`jira_specialist` has no source.** Black-box only, single probe, transcript captured, smoke assertions only per AC19.
6. **StrictMode double-fire on the Hosted UI test.** Mitigated by using programmatic login for 4 of 5 E2E spec files. The one Hosted UI spec runs against the deployed prod build (not StrictMode), so the issue does not surface.
7. **`MODEL_PRICING` drift between the two source files.** Already mitigated by AC23 mechanism in `pricing.py`. The risk is the harness panics on a legitimate single-file edit during a model add. Mitigation: the error message names both files and tells the user to edit both, matching the project rule.
8. **Off-limits files.** The implementer must not touch `Infra/params/dev.json`, `09-agentcore.yaml`, `buildspec.yml`, the two `MODEL_PRICING` constants, or any agent source. The harness is read-only against all of them. This is enforced socially (in this plan and the README) and mechanically (the harness has no boto3 calls into cloudformation, iam, kms, or ssm).

## 7. Test approach for the harness itself

The harness ships its own smoke tests under `tests-adversarial/tests/` so the implementer can verify each module without invoking the deployed env:

- `test_pricing.py` — fixtures with synthetic `agents/_shared/token_usage.py` and `ui/src/mockData.js` strings, asserts `load_pricing()` returns the merged dict on match, raises `PricingDriftError` on mismatch.
- `test_preflight.py` — asserts `enforce()` raises `RuntimeError` when estimated > cap; passes when under.
- `test_tracker.py` — asserts `add()` increments per-layer totals and `increment_probe()` raises after 30.
- `test_cognito_auth.py` — mocks `boto3.client("cognito-idp")` via `moto`, asserts `fetch_token("ciso")` returns the shape and `DEMO_PASSWORD` unset raises early.
- `test_builder.py` — synthetic manifest + synthetic layer result, asserts matrix has expected cells.
- `test_renderer.py` — golden-file test: feeds renderer a fixed manifest + results dict, diffs the produced HTML/MD against a checked-in golden.
- `test_diff.py` — synthetic current + baseline, asserts diff shape matches spec §6.3.

These run via `pytest tests-adversarial/tests/` and do **not** hit the deployed env. They are the developer-loop signal during harness build.

## 8. Out of scope (explicit)

Implementer must not:

- Add load testing (k6, Locust, Distributed Load Testing).
- Add chaos engineering (AWS FIS, runtime kills, DDB throttling).
- Add visual regression or pixel diffing.
- Add accessibility testing (axe-core sweep). `tests-e2e/accessibility.spec.ts` stays as-is.
- Modify `Infra/params/dev.json`, `Infra/templates/09-agentcore.yaml`, `buildspec.yml`.
- Modify `MODEL_PRICING` in `agents/_shared/token_usage.py` or `ui/src/mockData.js`.
- Modify any `agents/*/agent.py` or `Infra/functions/api_handler/api_handler.py`.
- Add CI gating (no GitHub Actions / CodeBuild changes).
- Wire PR comments. Out of scope for v1 per spec §2.
- Spin up scratch environments. Harness targets deployed dev only.
- Probe `jira_specialist` beyond a single black-box prompt.
